## Threading issue 
-after adding the GUI window that will pop up when hotkey is clicked in overlay_winodw and tried to run program, 
what I didnt know was both libaries (pynput, PyQt) ran 2 threads causing a multi thread,

-pynput internally runs its own background thread in the  when we estanlish keyboard.listner //waits for custom hotkey to be inserted 
- PyQt6 runs its own thread when the window UI pops up creating a UI thread 
now we have 2 threads running if the background thread touches UI then the program can crash 


FIX:    allowing Input Monitoring permission to VS Code is what allows pynput to listen globally
BEFORE: pynput → tries to hook keyboard → macOS blocks → crash
AFTER:  pynput → hooks keyboard → macOS allows → listener works


macOS restricts access to global input events for security reasons. Without explicit permission, attempts to create an event tap result in a system-level exception (NSException), terminating the process.
This issue was identified by analyzing the crash stack trace, which referenced macOS frameworks (SkyLight, CoreFoundation) responsible for input event handling.
The problem was resolved by enabling Input Monitoring permissions for the execution environment (VS Code / Terminal), allowing pynput to successfully register and listen for global keyboard events.

## GUI only hsowing on desktop home page, not on browsers/ differnt processes 

when presssing the hotkey, the GUI would run on the listenrs thread, whtich ran in the background not on main thread,
macOS AppKit exepcts this thread to run on main, therefore running on a backgorund thread insted of the main thread is why we coudlnt see the the GUI 
So instead of calling overlay.toggle() directly from the hotkey, main.py queues that call onto the main queue:
'''
main.py
Lines 19-26
    overlay = Overlay.alloc().init()
    main_queue = NSOperationQueue.mainQueue()
    def schedule_toggle():
        main_queue.addOperationWithBlock_(lambda: overlay.toggle())
    listener = CustomKeyListener(schedule_toggle)
'''
